看起來有點複雜，但大家就記得一件事情：我們常說你要搞清楚三個角色——

你在 GitHub 上的那個 repository
你 local 會有一個，等於是 local 的版本控制中心，就是你 local 的 repository
你把這個 repository checkout 出來的你的 working area

至少你要了解這三個的關係是什麼。
當然上禮拜有稍微講一下，你可能可以從別人的 repository fork 一個專案過來，這是另外一件事，大家稍微知道一下就好。

好，所以這三個的關係，我們就只要 focus 在這幾條線就好了。
大家知道，從 GitHub 到 local repository 到你自己，沒有，就是要透過這個「沒有」——有一條線是這樣過來的。
所以你知道從 remote 的 repository 過來，基本上就是拉到、把 code 同步到你的 local repository，主要就兩種：

一個是 fetch
一個是 pull

我也跟大家講過差別：fetch 就是把 code 拉下來而已；pull 除了拉下來之外，還幫你把 code merge 到你的 working directory 底下，所以稍有不同。
一般我們會建議大家用 fetch，後面也會講到，可能會用 rebase 來做一個同步，而不是直接 merge。
所以這個關係其實很單純，就是 fetch → rebase，然後你自己再去……
我們知道一個專案有很多 branch，你可能做好不同的 branch，讓你可以做一些平行的開發或管理，所以你是用 checkout 來切換/建立一個 branch，大致上是這樣。
那寫完之後要 push 回去，就是 git push，這邊可能就會有 rebase 跟 merge 這些指令。
所以 hopefully 大家了解這個概念之後，這邊有個小圖，就是為什麼要 rebase 而不是用 git merge？

那這些指令很快念過一遍，希望大家都知道意思：

git fetch upstream abc
（這個 upstream 另外補充：你通常會設兩個 remote，一個 origin 指到你自己的 repository，upstream 會指到你 fork 的那個 repository。
所以 git fetch upstream abc 的意思是：我想把別人持續改版的最新進度拉下來。）
